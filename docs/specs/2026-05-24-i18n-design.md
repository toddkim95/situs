# Situs i18n Design

## Goal

Situs should be usable by global open-source users without making the CLI
larger or harder to maintain. English remains the default language. The first
localized languages are Korean and Simplified Chinese.

## Initial Locales

- `en`: default and fallback
- `ko`: Korean
- `zh-Hans`: Simplified Chinese

Future likely additions:

- `ja`: Japanese
- `zh-Hant`: Traditional Chinese
- `es`: Spanish

Do not add more languages until the core message system is stable and existing
translations are covered by tests.

## Scope

Translate user-facing runtime text:

- picker header, footer, help, inspect labels, empty/loading messages
- `situs keymap`
- `situs setup`
- `situs doctor`
- top-level help and common error messages
- troubleshooting hints printed by CLI commands

Do not translate protocol or machine-readable values:

- commands and flags such as `choose`, `--command`, `--picker`
- config keys and environment variables such as `SITUS_PICKER`
- widget protocol actions such as `run` and `cd`
- history source values such as `local` and `atuin`
- file paths, shell snippets, and command examples

Documentation remains English-first. Korean docs can be added as
`docs/ko/`, but Chinese docs should wait until the runtime UI is stable and a
human review path exists.

## Locale Resolution

Locale is resolved in this order:

1. `SITUS_LANG`
2. `LC_ALL`
3. `LC_MESSAGES`
4. `LANG`
5. fallback `en`

Accepted aliases:

- `en`, `en_US`, `C`, `POSIX` -> `en`
- `ko`, `ko_KR` -> `ko`
- `zh`, `zh_CN`, `zh_Hans`, `zh-Hans` -> `zh-Hans`
- `zh_TW`, `zh_HK`, `zh_Hant`, `zh-Hant` should fall back to `en` until
  `zh-Hant` is explicitly supported

If `SITUS_LANG` is set to an unsupported value, print a short warning only for
interactive commands and fall back to `en`. Non-interactive protocol commands
must stay quiet.

## Architecture

Add an `i18n` module with three responsibilities:

- parse environment locale into a `Locale`
- expose typed message keys
- return localized strings with English fallback

Preferred shape:

```rust
pub(crate) enum Locale {
    En,
    Ko,
    ZhHans,
}

pub(crate) struct I18n {
    locale: Locale,
}

impl I18n {
    pub(crate) fn from_env() -> Self;
    pub(crate) fn text(&self, key: MessageKey) -> &'static str;
    pub(crate) fn format_count(&self, singular: MessageKey, plural: MessageKey, count: usize) -> String;
}
```

Use typed keys instead of raw string IDs so missing messages are caught by the
compiler where possible. Dynamic messages should keep variables outside the
translation when possible:

```rust
format!("{} {count}", i18n.text(MessageKey::HistoryRecords))
```

If a sentence must reorder variables across languages, add a small formatting
helper for that exact message instead of using ad-hoc string concatenation
throughout the codebase.

## Picker UX Rules

The picker must remain compact. Translations should not make rows wrap or break
the fixed bottom query line.

Rules:

- footer shortcuts may use English key names but localized action words
- if translated help is too wide, use the existing compact footer fallback
- row columns must continue to prioritize command and directory readability
- inspect/help views can use fuller translated descriptions
- tests must include at least one wide-character locale case, using Korean or
  Chinese text, because terminal display width matters

## Testing

Unit tests:

- locale resolution from `SITUS_LANG`, `LC_ALL`, `LC_MESSAGES`, `LANG`
- unsupported locale fallback
- every `MessageKey` has an English value
- Korean and Chinese message tables have all required keys
- picker render output stays stable with Korean/Chinese labels
- no protocol values are localized

Acceptance tests:

- `SITUS_LANG=ko situs keymap`
- `SITUS_LANG=zh-Hans situs keymap`
- `SITUS_LANG=ko situs doctor`
- zsh/PTY picker smoke with `SITUS_LANG=ko`
- zsh/PTY picker smoke with `SITUS_LANG=zh-Hans`

Regression requirement:

- Any new user-facing message must add an English value and either localized
  Korean/Chinese values or an explicit fallback decision in tests.

## Rollout

1. Add the i18n module and locale resolution.
2. Move keymap/setup/doctor text first because they are easy to verify.
3. Move picker text next because it is width-sensitive.
4. Move help/error text after the core flow is stable.
5. Update README, configuration docs, troubleshooting docs, and AGENTS.md.

## Open Decisions

- Keep English command names and flags forever.
- Keep English as the only mandatory documentation language for now.
- Do not add runtime translation files yet. Static Rust tables are simpler for a
  small CLI and avoid packaging mistakes.
